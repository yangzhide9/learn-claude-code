  3. 工具注册表 (L91-107)

  - TOOL_HANDLERS：工具名 → 处理函数的映射
  - TOOLS：向 Claude API 声明工具的 JSON Schema，告诉模型每个工具的参数格式

  4. normalize_messages() (L110-169) — 核心新增

  这是 s02 最重要的新函数，做三件事：

  ① 清理内部元数据字段（以 _ 开头的 key）
  ② 为孤立的 tool_use 补充 placeholder tool_result（避免 API 报错）
  ③ 合并连续相同 role 的消息（API 要求 user/assistant 严格交替）

  5. agent_loop() (L172-190)

  与 s01 几乎完全相同，只多了两点：
  - 调用前先 normalize_messages() 清理历史
  - 处理 tool_use 类型的响应块：找到对应 handler 执行 → 输出结果 → 把结果作为 user
  消息追加回历史

  ---
  关键设计理念

  ▎ "The loop didn't change at all. I just added tools."

  Agent 循环的本质是：调 API → 如果需要工具就执行 → 把结果喂回去 → 再调
  API。工具的扩展不需要改变循环结构，只需要扩充 TOOL_HANDLERS 和 TOOLS 列表即可。

  Step 1：清理内部元数据字段 (L156-169)

  clean["content"] = [
      {k: v for k, v in block.items() if not k.startswith("_")}
      for block in msg["content"]
      if isinstance(block, dict)
  ]

  问题背景：程序内部有时会给消息块打上调试标记（比如 _debug_info、_timestamp），这些字段 Claude
  API 不认识，发过去会报错。

  做法：把每个 content block 里，所有以 _ 开头的 key 全部过滤掉，只保留干净的字段。

  原始 block:
  { "type": "text", "text": "hello", "_debug": "step3", "_ts": 1234 }

  清理后:
  { "type": "text", "text": "hello" }

  ---
  Step 2：为孤立的 tool_use 补充占位结果 (L171-201)

  理解这个问题，先要理解 Claude API 的协议：

  每当 Claude 调用一个工具，消息历史里会出现一对：

  assistant → tool_use  (id: "abc123", name: "bash", input: {...})
  user      → tool_result (tool_use_id: "abc123", content: "输出结果")

  API 强制要求：每个 tool_use 必须有一个对应的 tool_result，否则报错。

  什么情况会出现"孤立"的 tool_use？

  用户中途按 Ctrl+C 取消了工具执行，或者程序崩溃了。历史里有 tool_use，但没有对应的
  tool_result。

  处理流程：

  # 第一步：收集所有已存在的 tool_result ID
  existing_results = set()  # e.g. {"abc123", "def456"}

  # 第二步：遍历 assistant 消息，找出没有配对的 tool_use
  if block.get("type") == "tool_use" and block.get("id") not in existing_results:
      # 补一个占位 tool_result
      cleaned.append({
          "role": "user",
          "content": [{"type": "tool_result", "tool_use_id": block["id"], "content":
  "(cancelled)"}]
      })

  修复前（API 会报错）:
  assistant: [tool_use id="abc123"]
  （没有 tool_result）

  修复后（API 正常接受）:
  assistant: [tool_use id="abc123"]
  user:      [tool_result tool_use_id="abc123", content="(cancelled)"]

  ---
  Step 3：合并连续相同 role 的消息 (L203-218)

  问题背景：Claude API 要求消息必须严格交替 —— user → assistant → user → assistant...

  不允许出现：
  user: "帮我写代码"
  user: [tool_result: "bash 执行结果"]   ← 两条连续 user 消息，API 报错！

  什么时候会出现连续相同 role？

  Step 2 在末尾追加了 placeholder tool_result（role=user），但列表里前面可能已经有一条 user
  消息紧挨着它。

  处理做法：遍历时，如果当前消息和上一条 role 相同，就把 content 合并进去：

  if msg["role"] == merged[-1]["role"]:
      # 把当前消息的 content 拼接到前一条消息的 content 列表里
      prev["content"] = prev_c + curr_c
  else:
      merged.append(msg)

  合并前:
  user: "帮我写代码"
  user: [tool_result: "(cancelled)"]

  合并后（一条 user 消息，两个 content block）:
  user: ["帮我写代码", tool_result: "(cancelled)"]

  ---
  三步关系总结

  原始 messages
        │
        ▼ Step 1
    清理 _ 字段（去掉 API 不认识的内部标记）
        │
        ▼ Step 2
    补全孤立 tool_use（防止 API 因缺少 tool_result 报错）
        │
        ▼ Step 3
    合并连续同 role（满足 API 的严格交替要求）
        │
        ▼
    可以安全发给 API 的 messages

  这个函数的本质是防御性清理——让 messages
  历史无论经历了什么意外（取消、崩溃、内部标记污染），都能安全地发给 API。
